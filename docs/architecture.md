# 系统架构

## 1. 分层设计（最重要的一张图）

```
┌─────────────────────────── 树莓派（本地·常开·免费）───────────────────────────┐
│                                                                              │
│  柱子摄像头 ──> 姿态估计(MediaPipe) ──> 坐姿判定(纯几何) ──> 节流器 ──┐        │
│  灯头摄像头 ──> 手部估计 ──────────────> 握姿判定 ───────────────────┤        │
│                                                                     ▼        │
│                                                        低速率反馈 UI / 语音    │
│                                                                     │        │
│                              （只有"作业"这种事件，才往上捅）          │        │
└─────────────────────────────────────────────────────────────────────┼────────┘
                                                                       │ 事件触发
┌──────────────────────────── 云端 / 自建 GPU（按需·偶发）──────────────▼────────┐
│   作业题理解(VLM)  ──>  苏格拉底式引导对话  ──>  错误归类 / 知识点打标签         │
│                                              ──>  周期性「能力周报」生成        │
└──────────────────────────────────────────────────────────────────────────────┘
                                         │
                                         ▼
                          本地 SQLite（错题集 / 能力画像，数据永远属于你）
```

**为什么这样分：**
- 坐姿/握笔是**连续监测**：放本地，免费、即时、私密、断网可用。
- 作业引导需要**语言理解**：交给大模型，但它是**事件触发、偶发**的，所以 token 成本很低。
- 把「数据」（SQLite 里的结构化事实）和「LLM 生成的视图」（周报）分开——
  换模型/换厂商，错题集原封不动。

## 3. 关键工程约定

1. **算法与 IO 分离**：判定逻辑是纯函数（标准库即可），摄像头/MediaPipe/屏幕
   都在外圈。好处：在没有摄像头的开发服务器上也能跑全部单测，逻辑改动有回归保障。
   —— 类比：DSP 内核独立于驱动。
2. **相对而非绝对**：坐姿判定永远相对于「这孩子的标定基准」，不写死绝对角度。
   换机位、换小孩，只需重新标定 10 秒，零改代码。
3. **配置外置**：阈值放 `config/posture.yaml`，调参不动代码。
4. **数据早记录**：一旦作业模块开工，第一件事就是建 SQLite 表开始记录——
   没记录过的历史无法回溯采集。

## 4. 硬件路线（详见 docs/hardware.md）

- 原型：手上的 **Pi 3**（单 USB 摄像头，跑 Stage 0~1 够用）。
- 量产形态：**Pi 5 (8GB)**，双摄（柱子 CSI 看坐姿 + 灯头 USB 看作业）。
- 作业引导：先用云端 API（豆包）验证交互；后续可自建 Qwen-VL 到 GPU 服务器。
